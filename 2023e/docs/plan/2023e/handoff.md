# 2023 E 题方案交接文档

## 当前目标

题目：运动目标控制与自动追踪系统。

当前阶段不是写代码，而是先完成算法建模和系统方案设计。用户倾向使用 OpenCV，硬件上使用固定摄像头、树莓派/香橙派、步进电机云台，并希望先讨论算法层，驱动层只保留通信接口。

## 已确认约束

1. 红色系统和绿色系统采用两套完全独立硬件。
2. 两套系统之间不通信。
3. 摄像头固定安装，不随云台转动。
4. 视觉算法运行在树莓派/香橙派上。
5. 云台执行器为步进电机。
6. 当前不讨论步进电机驱动细节，只设计算法层和驱动通信接口。
7. 接受每次设备摆放后的短标定流程。
8. 红色路径生成选择“识别四角生成几何路径 + 运行时视觉误差修正”的组合方案。

## 当前推荐建模方向

推荐不要把主方案建立在完整三维几何反解上，而是采用：

```text
屏幕平面二维坐标 + 相机标定/单应性 + 短标定 + 视觉伺服闭环
```

原因：比赛评判的是屏幕上的光斑位置误差和运动效果，而不是云台角度本身。固定摄像头和短标定可以把复杂机械误差、安装误差、激光笔偏心、云台轴不共点、步进电机回差等问题转化为闭环误差修正问题。

## 三种方案对比

### 方案 A：纯三维几何反解

使用世界坐标系、云台位置、高度、屏幕距离等参数，直接计算目标屏幕点对应的云台水平角和俯仰角。

优点：理论清晰，适合写报告。

缺点：对机械和安装误差敏感，实际精度风险高。

结论：可作为理论建模补充，不建议作为主控制方案。

### 方案 B：相机单应性 + 电机映射表

通过识别屏幕四角建立图像坐标到屏幕坐标的单应性变换，再短标定若干组“电机步数 -> 屏幕坐标”样本，用插值获得目标步数。

优点：比纯几何更抗安装误差，适合作为前馈定位。

缺点：如果机械回差明显或标定点不足，仍需闭环修正。

结论：适合作为初始定位或加速层。

### 方案 C：相机单应性 + 视觉伺服闭环

每帧检测当前光斑和目标点在屏幕平面内的位置，计算误差，再输出步进电机增量，持续修正。

```text
摄像头图像
-> 畸变矫正
-> 目标/光斑检测
-> 单应性变换到屏幕坐标
-> 计算误差
-> 控制器输出步进电机增量
-> 下一帧继续闭环修正
```

优点：最抗机械误差，适合比赛现场变化。

缺点：依赖光斑和目标检测稳定性，需要调好控制增益和限速。

结论：推荐作为主方案。

## 推荐系统架构

### 1. 视觉层

职责：从摄像头图像中提取屏幕坐标信息。

主要模块：

1. 相机畸变矫正。
2. 屏幕四角检测或人工辅助标定。
3. 图像坐标到屏幕坐标的单应性变换。
4. 红色光斑检测。
5. 绿色光斑检测。
6. A4 靶纸四角/黑色边框检测。

输出统一使用屏幕平面坐标，例如：

```text
ScreenPoint(x_mm, y_mm)
```

### 2. 任务层

职责：根据题目任务生成目标点或目标路径。

红色系统任务：

1. 基本要求 1：红点移动到屏幕中心。
2. 基本要求 2：红点沿屏幕四边顺时针移动。
3. 基本要求 3/4：识别 A4 靶纸四角，生成四边路径，红点沿路径运动。

绿色系统任务：

1. 发挥部分 1：识别红点和绿点，控制绿点追踪红点。
2. 发挥部分 2：红色系统独立沿 A4 路径运动，绿色系统独立视觉追踪红点。

### 3. 控制层

职责：把屏幕坐标误差转换成步进电机控制量。

推荐控制形式：

```text
error = target_screen_point - current_laser_point
delta_step_yaw = Kx * error_x + Dx * (error_x - last_error_x)
delta_step_pitch = Ky * error_y + Dy * (error_y - last_error_y)
```

实际可以先从 P 控制开始，再加限速、死区和简单 D 项。

建议机制：

1. 误差小于阈值时停止或低速微调。
2. 每帧最大步数增量限幅，避免过冲。
3. 光斑丢失时暂停输出，等待重新识别。
4. 目标点跳变时使用路径插值，避免云台突然大幅运动。

### 4. 驱动接口层

当前不实现具体驱动，只定义算法层输出接口。

建议接口：

```text
MotorCommand(
  yaw_delta_steps,
  pitch_delta_steps,
  max_speed,
  command_id
)
```

或：

```text
MotorTarget(
  yaw_target_steps,
  pitch_target_steps,
  max_speed,
  command_id
)
```

推荐优先使用增量步数接口，因为视觉闭环天然每帧输出修正量。

## 坐标系建议

当前 `plan_v0.md` 中定义了三维世界坐标系。后续建议保留三维坐标用于报告说明，但算法主流程使用二维屏幕坐标系。

建议屏幕坐标系：

```text
屏幕左下角：ScreenPoint(0, 0)
屏幕右下角：ScreenPoint(500, 0)
屏幕左上角：ScreenPoint(0, 500)
屏幕右上角：ScreenPoint(500, 500)
单位：mm
```

这样题目中的 2cm、3cm、4cm 误差可以直接换算为 20mm、30mm、40mm。

## 标定流程建议

### 上电/摆放后短标定

1. 固定摄像头拍摄屏幕。
2. 检测或手动确认屏幕四角。
3. 计算图像坐标到屏幕坐标的单应性矩阵。
4. 控制红色激光移动到若干采样点，记录步数和屏幕坐标。
5. 控制绿色激光移动到若干采样点，记录步数和屏幕坐标。
6. 得到用于初始定位的粗映射。
7. 后续运行依靠视觉闭环修正最终误差。

### 相机内参标定

建议赛前完成，不必每次现场做。

使用棋盘格/圆点板标定相机内参和畸变参数，运行时先做畸变矫正，再做屏幕单应性。

## 红色系统建模

红色系统不需要知道绿色系统存在。

输入：

```text
camera_frame
task_mode
```

输出：

```text
MotorCommand for red gimbal
```

核心闭环：

```text
target_path_point(t) - current_red_laser_point -> motor_delta_steps
```

路径生成：

1. 屏幕四边任务：直接使用屏幕四角生成闭合矩形路径。
2. A4 靶纸任务：识别 A4 四角后，按顺时针顺序生成闭合四边路径。
3. 路径运动时，不直接跳到下一个角点，而是按固定步长生成连续目标点。

## 绿色系统建模

绿色系统不能读取红色系统的路径、状态或指令。

输入：

```text
camera_frame
```

输出：

```text
MotorCommand for green gimbal
```

核心闭环：

```text
detected_red_laser_point - current_green_laser_point -> motor_delta_steps
```

这符合题目“绿色光斑自动追踪红色光斑”的要求，也符合两系统独立、不通信的约束。

## 目标识别建议

### 光斑检测

优先使用 HSV 颜色阈值 + 亮度阈值 + 连通域/轮廓中心。

红点：红色 HSV 需要处理 hue 在 0/180 两端断裂的问题。

绿点：绿色 HSV 阈值相对简单。

建议输出：

```text
LaserDetection(
  found,
  image_center,
  screen_point,
  area,
  confidence
)
```

### 屏幕/A4 检测

屏幕四角：可用边框、亮度差、人工点击辅助或 ArUco/标记点辅助。

A4 靶纸：可检测黑色边框/黑色胶带轮廓，再用多边形拟合得到四角。

如果现场背景复杂，建议在 A4 靶纸四角加入高对比度辅助标记，但要确认是否符合题目规则。

1. 红色光斑沿路径运动时，速度指标如何设定：优先稳定低误差，还是优先速度？
2. 屏幕四角和 A4 四角是否允许人工点击辅助标定？
3. 两套系统是否各有一块屏幕视野完整的摄像头？
4. 光照环境是否可控，是否需要滤光片或降低曝光来增强光斑检测？
5. 控制接口最终使用增量步数还是绝对目标步数？
